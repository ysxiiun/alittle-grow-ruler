## 通用编码原则（所有语言适用）

### 【强制】

- **命名清晰**：禁止使用拼音和无意义的 a/b/temp
- **单一职责**：函数/方法不宜过长（建议不超过 80 行）
- **禁止循环内耗时操作**：数据库查询、远程调用等移出循环
- **禁止魔法值**：必须定义为常量或枚举
- **文档完整**：所有公开接口必须有文档说明
- **错误处理**：必须处理，禁止空 catch/except 块
- **敏感信息**：禁止在日志/代码中硬编码密码、密钥

### 【推荐】

- **DRY 原则**：相同逻辑不重复超过 3 次
- **条件清晰**：复杂条件提取为语义清晰的布尔方法
- **空安全**：优先使用语言原生的空安全特性
- **日志规范**：关键节点必须有日志，异常必须有 ERROR 日志

## TypeScript / React / Node.js 专属规范

### 命名规范【强制】

- 变量、函数、Hook 使用 `camelCase`
- React 组件、页面目录使用 `PascalCase`
- 类型、接口、枚举使用 `PascalCase`
- 服务端路由、数据库模块按资源语义命名，如 `records.ts`、`analysis.ts`

### 代码结构【强制】

- 保持前端 `pages`、`components`、`hooks`、`utils` 分层清晰
- 保持后端 `routes`、`database`、`templates`、`mcp` 分层清晰
- 单个函数职责聚焦，复杂图表转换逻辑优先抽成局部辅助函数
- 默认使用显式类型，避免让核心数据结构退化为 `any`

### 注释规范【强制】

- 公开接口、复杂计算逻辑、模板驱动行为保留简短注释
- 注释说明原因和约束，不写低价值逐行解释
- 中文文案保持统一，新增文案优先沿用已有表达

### 异常/错误处理【强制】

- 前端请求失败必须有用户可见反馈
- 服务端路由必须返回明确状态码和错误信息
- 禁止吞掉异常；捕获后至少记录日志或返回错误响应

### 日志规范【强制】

- 服务端启动、数据库初始化、模板初始化等关键节点保留日志
- 仅在排障需要时增加日志，避免噪声日志污染输出
