# 修复孕期体重数据日期保存

> 序号：004
> 日期：2026-04-26 22:06
> 任务类型：Bug 修复

## 任务概述

修复孕期体重记录在凌晨 `0:00-3:00` 场景下，表单显示的数据日期与最终保存结果不一致的问题，并确保用户手动修改数据日期后不会被后续表单联动覆盖。后续排查发现生产端口曾运行旧构建且旧进程持有的 SQLite 文件与当前磁盘文件不一致，已从 Chrome API 缓存恢复 `record_id=3` 的 198 条孕期体重数据。

## 技术方案摘要

将孕期体重表单中“体重类型是否手动修改”和“数据日期是否手动修改”的标记改为 `useRef` 管理，避免 React state 在 `onValuesChange` 中读取到旧值。保存前若数据日期未被用户手动修改，则再次按记录时间与体重类型计算数据日期；若已手动修改，则尊重用户输入。

## 变更内容

| 文件 | 变更类型 | 说明 |
|---|---|---|
| `src/client/pages/DataForm/index.tsx` | 修改 | 修复孕期体重数据日期联动与保存逻辑，确保自动规则与手动覆盖都生效 |
| `data/grow-ruler.db` | 运行时恢复 | 从 Chrome 缓存恢复孕期体重记录集 198 条数据，并按凌晨睡前规则补齐 `data_date` |
| `dist/` | 构建 | 执行 `npm run build` 后刷新生产构建产物 |

## 遇到的问题与解决方案

当前源码 API 直接写入和更新 `data_date` 均可正常落库，前端联动状态也需要修复。原实现使用 state 判断用户是否触碰过数据日期，但 `setFieldValue` 与 `onValuesChange` 的交互容易留下过期标记，导致后续联动误判。改为 ref 后，联动判断即时生效，并在提交前增加一次业务规则兜底计算。

生产排查中，旧服务进程曾返回 `record_id=3`、`total=198`，最近数据到 `2026-04-26 10:36:46`；停止旧进程后，当前磁盘 `data/grow-ruler.db` 是空库。通过 Chrome Cache 中的 `/api/records?record_id=3` 响应恢复了 id `4-201` 的 198 条数据，其中 id `200` 的 `timestamp=2026-04-26 01:37:02`、`weight_period=night`，恢复后的 `data_date=2026-04-25`。

## 技术决策记录

保留后端 `data_date` 入参优先的接口契约，因为编辑场景需要允许用户主动指定数据日期；本次仅修复前端表单状态与提交计算，不调整数据库结构或 API 契约。
